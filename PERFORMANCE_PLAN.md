# خطة تحسين أداء النظام (Back-End + Templates + Front-End)

هذه الخطة ليست تحسينات عشوائية لقالب أو Route منفرد، بل مسار متسلسل مبني على القياس، يضمن عدم تكرار نفس الأخطاء ويعالج النظام كمنظومة واحدة.

## المبادئ

- القياس قبل التعديل وبعده (قبل/بعد) هو المعيار.
- أي تحسين يجب أن يقلّل: زمن الطلب، زمن الـSQL، عدد الاستعلامات، أو زمن رندر القالب، أو حمل DOM/JS.
- يمنع إدخال عمليات ثقيلة داخل صفحات الاستعراض (قوائم/تقارير) مثل إعادة حساب أرصدة أو إصلاحات تلقائية.
- التغييرات تُنفّذ على شكل دفعات صغيرة قابلة للعكس، مع اختبارات وLint بعد كل دفعة.

## المرحلة 1: القياس والمراقبة (Observability)

**الهدف:** معرفة أين يضيع الوقت بالضبط بدل الاعتماد على الانطباع.

- إضافة قياس موحّد لكل Request:
  - `total_ms`: زمن الطلب الكامل
  - `sql_ms`: زمن SQL التراكمي
  - `sql_count`: عدد الاستعلامات
  - `template_ms`: زمن رندر القالب (إن وجد)
  - `endpoint` / `path` / `method` / `status`
- تخزين آخر N سجل (مثلاً 1000) في ذاكرة العملية أو Cache.
- عرض صفحة داخلية (Admin/Dev) لقراءة النتائج:
  - أبطأ الصفحات حسب `total_ms`
  - أكثر الصفحات SQL-heavy حسب `sql_ms` و `sql_count`
  - فلترة حسب endpoint أو path
- وضع مفتاح تشغيل/إيقاف للقياس (config/setting) لضمان عدم إضافة حمل دائم في الإنتاج عند عدم الحاجة.

**مخرجات المرحلة 1:**
- تقرير داخلي (صفحة) يظهر Top Slow Pages.
- معيار ثابت للمقارنة قبل/بعد.

## المرحلة 2: حصر أبطأ 20 صفحة وتصنيف سبب البطء

**الهدف:** تحويل “البطء” إلى قائمة مهام واضحة.

- استخراج Top 20 endpoint حسب:
  - `total_ms`
  - `sql_ms`
  - `sql_count`
  - `template_ms`
- تصنيف كل صفحة إلى سبب/أسباب:
  - DB: استعلامات كثيرة أو Join ثقيل أو عدم وجود Index
  - Template: حلقات Jinja على آلاف العناصر أو فلترة/Map داخل القالب
  - Frontend: DataTables/Select2 على DOM ضخم أو إعادة init متكررة
  - Assets: تحميل مكتبات ثقيلة في base.html لكل الصفحات
  - Side-effects: إعادة حسابات/تصحيحات داخل request

**مخرجات المرحلة 2:**
- قائمة صفحات + سبب أساسي + “حل نمطي” لكل سبب.

## المرحلة 3: توحيد نمط Pagination للقوائم (System-wide)

**الهدف:** منع تحميل آلاف العناصر دفعة واحدة (سيرفر + متصفح).

- اعتماد نمط واحد في كل القوائم:
  - `page` و `per_page` مع سقف منطقي
  - `query_args` للحفاظ على الفلاتر عند التنقل
  - Pagination UI موحّد في القوالب
- تحويل القوائم الأكبر تدريجياً:
  - الموردين، الشركاء، العملاء، المنتجات، التحويلات، الشحنات، النفقات، السجلات
- منع DataTables تلقائياً على الجداول الضخمة (إلا عند force) أو التحويل إلى Pagination/Server-side.

**مخرجات المرحلة 3:**
- صفحات الاستعراض تصبح سريعة وثابتة مهما كبر حجم البيانات.

## المرحلة 4: تخفيف base.html والسيدبار وتحميل السكربتات شرطياً

**الهدف:** كل صفحة تستفيد، وليس صفحة واحدة.

- تقليل تحميل المكتبات الثقيلة في كل الصفحات (Charts/DataTables) وجعلها شرطية حسب الحاجة.
- مراجعة السيدبار والـcontext processors:
  - ضمان أن صلاحيات المستخدم/flags تُحسب مرة واحدة لكل request (cached per-request).
  - منع أي استعلام DB داخل Jinja globals التي تُستدعى كثيراً.

**مخرجات المرحلة 4:**
- تقليل زمن تحميل الواجهة ووقت تنفيذ JS في كل الصفحات.

## المرحلة 5: نقل إعادة حساب الأرصدة إلى مسار صريح/خلفي

**الهدف:** فصل “عرض البيانات” عن “صيانة البيانات”.

- منع أي Route استعراض يقوم بـ:
  - `refresh` لكل سجل
  - بناء breakdown لكل entity داخل loop
  - تحديث أرصدة أثناء العرض
- بدلاً من ذلك:
  - زر/Endpoint صريح “تحديث أرصدة” (مقيد بصلاحية)
  - أو مهمة دورية/خلفية (حسب بنية النظام)

**مخرجات المرحلة 5:**
- الاستعراض يصبح سريع دائماً، والصيانة تتم بشكل متحكم.

## المرحلة 6: تحسين قاعدة البيانات (Indexes)

**الهدف:** علاج جذري للصفحات DB-heavy.

- بناء قائمة بالحقول الأكثر استخداماً في الفلاتر والترتيب من القياس:
  - أمثلة: `(customer_id, sale_date)`, `(supplier_id, payment_date)`, `(partner_id, created_at)` …
- إضافة فهارس مناسبة بعد التأكد من أن الاستعلامات تعتمدها فعلياً.

## المرحلة 7: تحسينات JS عامة

**الهدف:** منع تجميد المتصفح عند التعامل مع DOM كبير.

- منع إعادة تهيئة شاملة عند أي تغيير DOM (init جزئي مقيّد).
- DataTables: تفعيل حسب حجم الجدول أو حسب `data-dt-force`.
- Select2: تفعيل AJAX للبحث بدل تحميل آلاف الخيارات.

## المرحلة 8: التحقق المستمر

**الهدف:** لا رجوع لنقطة الصفر.

- بعد كل دفعة:
  - تشغيل tests + lint
  - مقارنة Top Slow Pages قبل/بعد
  - توثيق الصفحات التي تحسّنت بالأرقام

---

## التسلسل التنفيذي الذي سنلتزم به

1. تنفيذ المرحلة 1 بالكامل (قياس + صفحة عرض النتائج).
2. استخراج Top Slow Pages (المرحلة 2) وتحديد أول 5 صفحات.
3. تطبيق نمط Pagination والحد من القوائم الثقيلة (المرحلة 3) على الصفحات الخمس.
4. تخفيف base.html والسيدبار (المرحلة 4) وإزالة الأحمال العامة.
5. نقل صيانة الأرصدة لمسار صريح (المرحلة 5).
6. إضافة Indexes بناءً على القياس (المرحلة 6).
7. تحسينات JS العامة (المرحلة 7).
8. توثيق النتائج بالقياس + اختبارات (المرحلة 8).

---

## حالة التنفيذ الحالية

- المرحلة 1: مكتملة
- المرحلة 2: جاهزة (تجميعة + تصنيف تلقائي) وتحتاج بيانات استخدام فعلية
- المرحلة 3: مطبّقة على عدة قوائم كبيرة (الشركاء/المنتجات/التحويلات/طلبات الأونلاين)
- المرحلة 4: مطبّقة (تحميل سكربتات شرطياً)
- المرحلة 5: مطبّقة (إلغاء auto-fix داخل صفحات العرض + Endpoints صريحة للتحديث)
- المرحلة 6: جاهزة للتطبيق عبر `flask optimize-db` بعد جمع Top DB-heavy
- المرحلة 7: مطبّقة جزئياً (Select2 lazy + تقليل حمل DataTables server-side)
- المرحلة 8: مستمرة (اختبارات + lint بعد كل دفعة)

## كيف نكمل المرحلة 2 بالأرقام

- افتح صفحة: `/system/performance`
- استخدم جدول "تجميعة" لتحديد Top 20 حسب:
  - `p95_total_ms` للبطء العام
  - `p95_sql_ms` و `p95_sql_count` لمشاكل قاعدة البيانات
  - `p95_template_ms` لمشاكل القوالب
- استخدم `/system/performance/snapshot` لحفظ Snapshot JSON قابل للمشاركة/الأرشفة
- إذا أردت حفظ سجل دائم (JSONL) على القرص داخل نفس عملية السيرفر:
  - اضبط `PERF_MONITOR_PERSIST_PATH` في إعدادات Flask
  - بعدها نفّذ: `flask perf-snapshot --out <path>.json` لتحليل الملف

## كيف نطبق المرحلة 6 (Indexes) بأمان

- نفّذ من CLI:
  - `flask optimize-db`
- الأمر يبني فهارس شائعة الاستخدام باستخدام `CREATE INDEX IF NOT EXISTS` ويحاول `ANALYZE` إن كان مدعوماً
